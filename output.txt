Backend Request
Please ensure live transfer notifications are sent through the backend whenever a public chat session enters the human-waiting/escalated state.

Required behavior:

When a chat user requests a human, the backend should transition the session into the transfer/escalated state.

At the same time, send SMS notifications to eligible support users.

Eligible users should be filtered by:

tenant/customer access
product access
department/routing match
active support user status
notification preferences, especially sms !== false
valid phone/mobile number
availability rules, if routing requires available agents
SMS should be sent once per transfer request, not repeatedly from polling.

The SMS should include enough context to act:

visitor name if known
product/customer
urgency if available
short message/inquiry summary
link to support console/session if available
Example SMS:

Workside Support: Jerry is waiting for human support in Merxus AI. Topic: pricing question. Open Support Console to accept the transfer.
Record notification results on the session so the support console can display them.

session.notificationStatus

session.support.notificationTimeline
session.support.routing.notificationStatus
session.support.routing.notificationTimeline

Customer Transcript On End Chat
Separately, the customer-facing “email me a copy” flow is not the same as the support console Send Transcript action.

Please add or confirm a public/customer-safe backend endpoint for the website chat widget, for example:

POST /public/chat/sessions/:sessionId/transcript
or whatever public chat route naming matches the backend.

Requirements:

Must not require support-user auth.
Must only send to the captured customer email for that session, or require a verified/same-session email.
Must use customer-safe defaults:
include visitor/customer messages
include human support messages
exclude internal notes
exclude system events
exclude private routing/admin details
AI messages optional, but default should be customer-safe
Should reject if session/email does not match.
Should rate-limit to prevent abuse.
Should record transcript receipt on the session.
Once that endpoint exists, the Merxus public chat widget can ask at chat end:

Would you like us to email you a copy of this chat?
and call that public transcript endpoint if the user confirms.
