Goal
When a visitor requests a human, backend should decide whether an available support person exists for the request type, assign or queue the session accordingly, notify the right people, and return clear state to the support console.

Departments
Create/standardize these initial departments:

sales
support
billing
general
Department model:

{
  "id": "support",
  "label": "Support",
  "product": "merxus",
  "active": true,
  "defaultAssigneeIds": [],
  "fallbackDepartmentId": "general",
  "createdAt": "...",
  "updatedAt": "..."
}
Prevent duplicate department IDs per product.

Support User Model
Extend support users with availability and notifications:

{
  "id": "firebase_uid_or_internal_id",
  "name": "Mike",
  "email": "mike@example.com",
  "phone": "+16613451154",
  "role": "support_agent",
  "active": true,
  "departments": ["support", "general"],
  "allowedProducts": ["merxus"],
  "allowedTenantIds": ["*"],
  "availability": {
    "status": "available",
    "manualOverride": false,
    "quietUntil": null,
    "lastSeenAt": "2026-05-02T17:00:00.000Z",
    "updatedAt": "2026-05-02T17:00:00.000Z"
  },
  "notificationPreferences": {
    "sms": true,
    "email": true,
    "consoleSound": true,
    "adminEscalations": true
  }
}
Availability statuses:

available
busy
away
offline
do_not_disturb
A user is available only if:

active === true
matching department
matching product
matching tenant access
availability.status === "available"
quietUntil is null or expired
either manualOverride === true or lastSeenAt is recent enough
Recommended recent threshold:

lastSeenAt within 2 minutes

Intent Classification
When visitor requests a person, classify intent into:

sales
support
billing
general
Use available structured hints first:

{
  "intent": "support"
}
Fallback keyword examples:

sales: pricing, demo, buy, plan, quote
billing: invoice, charge, payment, refund
support: help, broken, issue, problem, error
general: unknown/default
Persist:

{
  "intent": "support",
  "departmentId": "support"
}
Human Request / Transfer Route
When the visitor asks for a human, backend should:

Persist visitor message.
Capture lead/contact fields if supplied.
Classify intent.
Find matching department.
Find available support users.
Assign if available, otherwise queue.
Notify support/admin as appropriate.
Return updated session.
Canonical session update if available agent found:

{
  "status": "escalated",
  "transferRequested": true,
  "transferRequestedAt": "2026-05-02T17:00:00.000Z",
  "transferReason": "user_requested_human",
  "intent": "support",
  "departmentId": "support",
  "departmentLabel": "Support",
  "routingStatus": "assigned",
  "availabilityOutcome": "available_agent_found",
  "assignedToUserId": "user_id",
  "assignedToName": "Mike",
  "assignedToEmail": "mike@example.com"
}
If nobody is available:

{
  "status": "escalated",
  "transferRequested": true,
  "transferRequestedAt": "2026-05-02T17:00:00.000Z",
  "transferReason": "user_requested_human",
  "intent": "support",
  "departmentId": "support",
  "departmentLabel": "Support",
  "routingStatus": "queued_no_agent_available",
  "availabilityOutcome": "no_available_agent",
  "assignedToUserId": "",
  "assignedToName": "",
  "assignedToEmail": ""
}
Customer Chat Response
If agent is available:

I'm connecting you with someone now.
If no one is available:

No one is available right now, but someone will follow up shortly.
If contact info is missing:

Please leave your name and email so someone can follow up.
Notifications
When available agent found:

Notify assigned support user by SMS if enabled.
Notify console clients via realtime/poll state.
Optional email if enabled.
SMS example:

New Merxus support request from Jerry Seinfeld. Intent: Support. Open the support console to accept the transfer.
When no agent available:

Notify admins/supervisors unless muted.
Optional SMS/email based on preferences.
Admin SMS example:

Merxus support request is waiting with no available support agent. Intent: Support. Customer: Jerry Seinfeld.

Acceptance Criteria

Visitor requests human.
Backend persists status: "escalated" and transferRequested: true.
Backend classifies intent into sales/support/billing/general.
If available matching user exists, session is assigned to that user.
Assigned user receives SMS if enabled.
If no user is available, session remains queued and customer is told someone will follow up.
Admin receives notification unless muted.
GET /support/sessions shows routing status and availability outcome.
Console queue count increases for human-requested sessions.
Notification failure does not lose the transfer request.
