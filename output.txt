Backend Instructions

When a public chat session enters human-transfer/waiting state, send both email and SMS notifications to eligible support users.

Trigger points:

POST /chat/public/session/:id/request-human
any backend AI/escalation path that sets transfer state to transfer_requested, escalated, waiting_for_takeover, or equivalent
do not wait for support console polling
do not require the support console browser to be open
Eligibility:

Look up active support users with access to the session’s product, tenantId, and department/routing intent.
User must be effectively available:
availability = available
heartbeat not stale, unless backend policy allows SMS fallback even if stale
not muted/quiet/do-not-disturb
User must have a normalized E.164 phone number, for example +15555551212.
User must have SMS notifications enabled or no preference set if default is enabled.
Notification fan-out:

Send email as currently implemented.
Send SMS through the existing Merxus/Twilio SMS service at the same time.
SMS body should be short and contain no direct link for now:
Workside Support: New Merxus AI chat waiting for takeover from Jerry. Open the support console or mobile app to accept.
Include product/customer name if available, but avoid sensitive transcript content.
Persistence/audit contract:

Write notification attempts to session detail so the support console can show them:
notificationTimeline: [
  {
    channel: "sms",
    recipient: "+1 xxx-xxx-1234",
    recipientUserId: "support_user_id",
    status: "sent" | "failed" | "skipped",
    reason: "available_agent_found" | "missing_phone" | "sms_disabled" | "muted" | "provider_error",
    providerMessageId: "twilio_sid_if_available",
    attemptedAt: "2026-05-06T..."
  }
]
Also expose summary fields:
support: {
  routing: {
    notificationStatus: {
      attempted: true,
      channels: ["email", "sms"],
      channelResults: [...]
    }
  }
}
Failure handling:

Email success must not suppress SMS failure.
SMS failure must be visible in notificationTimeline.
Log provider errors with enough detail for operators, but do not expose secrets.
If no SMS is sent, backend should record a skipped reason such as missing_phone, sms_disabled, muted, no_available_agent, or invalid_phone.
Tests to add:

Available support user with phone receives email and SMS.
Email succeeds and SMS fails: session shows email sent, SMS failed.
Missing phone: SMS is skipped with missing_phone.
Muted user: SMS is skipped with muted.
Unavailable/stale user: SMS is not sent unless fallback policy says otherwise.
Notification timeline appears on GET /support/sessions/:id.
Also ask them to confirm that your support-user record has a valid phone value and SMS notifications enabled. That is the one frontend/admin-data thing that could explain “email yes, SMS no.”
